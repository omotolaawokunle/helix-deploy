# HelixDeploy — Design Context

Persistent design guidance for all UI work across the HelixDeploy monorepo (Vue 3 SPA + future marketing/docs surfaces).

---

## Design Context

### Users

**Who:** Platform engineers, DevOps leads, and senior developers who self-host HelixDeploy for their organization. They manage servers, deployments, pipelines, and production infrastructure — often under time pressure during incidents or release windows.

**Context:** Long sessions at a desk; switching between overview (health, status) and deep detail (logs, provisioning output, audit trails). Multi-tenant: they operate within one org at a time but need clear tenant boundaries.

**Job to be done:** Register and provision servers, deploy safely to production, diagnose failures from logs, and prove who changed what — without leaving the control plane.

**Emotional goal:** **Confidence and speed** — the UI should feel like a capable ops tool that gets out of the way, not a marketing site or a toy dashboard.

### Brand Personality

**Three words:** Confident · Minimal · Sharp

**Voice:** Direct, operational, no fluff. Labels name things plainly (`Provision Server`, not `Enhance your infrastructure journey`). Errors state what failed and what to do next.

**Reference feel:** [Laravel Forge](https://forge.laravel.com) / [Ploi](https://ploi.io) — familiar hosting control panels: sidebar navigation, server cards, status-first layouts, production warnings that cannot be missed.

**Anti-references:** Generic AI dashboards (purple gradients, cyan glow on dark, hero metrics with gradient text), cluttered “everything in a card” grids, playful consumer SaaS onboarding.

### Aesthetic Direction

| Dimension | Direction |
|-----------|-----------|
| **Visual tone** | Confident minimal SaaS with industrial ops clarity — Forge-like familiarity, sharper typography and spacing than legacy panels |
| **Theme** | **Light and dark equally supported** — user picks Light, Dark, or System in the account menu (`UserMenu`); persisted in `localStorage` (`helix-deploy-theme`) |
| **Color** | **Teal/cyan brand accent** on neutral OKLCH base (existing shadcn `neutral` scale). Teal signals infrastructure/deployment; red reserved for production/danger; semantic env badges unchanged |
| **Typography** | **Geist** (already configured via shadcn-vue `geist-sans`) — clean, modern, legible at small sizes in dense tables and log viewers |
| **Icons** | Lucide only — consistent 16–20px in nav, 24px max in empty states |
| **Motion** | Subtle: 150–200ms color/opacity transitions; pulse only for live/connecting states; respect `prefers-reduced-motion` |
| **Density** | Medium — breathable on index pages, tighter in log/terminal views |

### Design Principles

1. **Extend shadcn/vue, don’t replace it.** UI improvements go through `@/components/ui/*` primitives and shared `@/components/common/*` — variant tweaks, token updates, composition. Replace a layout only when information architecture is wrong, not for cosmetic drift.

2. **Tokens before one-offs.** Brand color, radius, and neutrals live in `apps/frontend/src/style.css` CSS variables (`--primary`, `--ring`, sidebar tokens). Feature pages use semantic classes (`bg-primary`, `text-muted-foreground`) — never hardcode hex in feature components.

3. **Status is the interface.** Server/deployment/daemon state must be scannable in <2s: badges, environment labels, production banners. Forge users expect this hierarchy.

4. **Progressive disclosure.** Index pages show summary; detail pages and sheets carry complexity (provision drawer, add server, log streams). Default views stay simple; advanced options expand inline.

5. **Production is visually distinct.** Production environments always trigger `ProductionWarningBanner` + stronger env badge treatment. Destructive actions use `ConfirmDestructiveDialog` — never inline-only delete.

6. **Logs feel like logs.** Terminal/provisioning views use dark monospace panels (`bg-zinc-950`, `font-mono`) — intentional exception to the light shell; not the default page chrome.

7. **No AI slop.** Avoid: gradient heroes, identical icon+title+card grids, gray-400 body text on white, decorative glass/blur, bounce easing, nested cards, centered everything.

### Implementation Constraints

- **Stack:** Vue 3 + TypeScript strict, Pinia, Vue Router, Tailwind v3, shadcn-vue (`reka-nova` style)
- **Component path:** `apps/frontend/src/components/ui/` — shadcn primitives; `components/common/` — domain-shared (badges, banners, dialogs)
- **Feature UI:** `apps/frontend/src/features/{domain}/` — pages compose shadcn + common components only
- **Theming:** `components.json` → `baseColor: neutral`; accent applied via `--primary` teal shift in `:root` / `.dark`
- **Accessibility:** WCAG 2.1 AA minimum — focus rings on shadcn components, labels on all form fields, 4.5:1 text contrast, keyboard nav for sidebar and sheets

### Teal Accent Tokens

Applied in `apps/frontend/src/style.css`:

```css
/* Light — illustrative; tune with OKLCH picker */
--primary: oklch(0.55 0.12 195);
--primary-foreground: oklch(0.99 0 0);
--ring: oklch(0.55 0.12 195);

/* Dark */
--primary: oklch(0.72 0.12 195);
--primary-foreground: oklch(0.15 0 0);
```

Sidebar active states and CTA buttons use `primary`; destructive stays `--destructive`.

### Page Patterns (Forge-aligned)

| Pattern | Guidance |
|---------|----------|
| **Index pages** | Left-aligned title + primary action right; grid or table — not hero + stats row |
| **Server cards** | Hostname prominent, IP monospace-muted, env badge top-right, status bottom-left |
| **Detail pages** | Header (name + status + actions) → tabs for domains → overview stats as simple rows or flat stat blocks, not 6 identical cards |
| **Sheets/drawers** | Add/provision flows — prefer `Sheet` side drawer over centered modals |
| **Empty states** | One icon, one headline, one action — teach the next step |
